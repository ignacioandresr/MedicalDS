COMO ESTE SOFTWARE CUMPLE LOS TIPOS DE PRUEBA

1. Unitarias (¿Qué se prueba?)
- Lógica interna de modelos (ej: Symptom, Patient, Diagnostic).
- Validaciones de campos y relaciones (síntoma pertenece a paciente).
- Métodos de cálculo / normalización (diagnóstico preliminar, traducción futura).
- Configuraciones de seguridad (cifrado, hashing) y utilidades de rendimiento.
Resultado esperado: Cada pieza funciona de forma aislada sin depender de otras capas.

2. Integración (¿Qué se prueba?)
- Flujo de registrar síntomas vía endpoint (auth + validación + persistencia).
- Generación de diagnósticos combinando BD + motor lógico/IA.
- Acceso a historial médico con permisos y posible exportación.
- Tiempo de respuesta del endpoint de diagnóstico (<3 segundos).
- Interacción con módulos especiales (traducción, simulación, offline) cuando existan.
Resultado esperado: Las capas (rutas, controladores, modelos, servicios, permisos) cooperan correctamente.

3. Sistema / End-to-End (¿Qué se prueba?)
- Escenarios completos: médico ingresa síntomas → obtiene diagnóstico rápido y seguro.
- Consulta de historial con control de acceso y exportación PDF.
- Uso de traducción alienígena aplicada a un diagnóstico (cuando esté implementada).
- Ejecución de simulaciones médicas interplanetarias validadas.
- Operación offline y sincronización posterior sin pérdida.
Resultado esperado: El sistema completo cumple flujos críticos como los usaría un usuario real.

4. Aceptación (¿Qué se prueba?)
- Cada requerimiento se valida contra su criterio (ej: diagnóstico <3s, historial sólo roles autorizados, traducción sin ambigüedad, offline sin pérdida de datos).
- Se documenta evidencia clara para el cliente (ID requerimiento → resultado). 
Resultado esperado: El cliente puede confirmar que se cumplió exactamente lo solicitado.

Resumen Beneficios
- Unitarias: Previenen errores básicos.
- Integración: Garantizan cohesión entre componentes.
- Sistema: Aseguran confiabilidad real en flujos completos.
- Aceptación: Certifican conformidad con expectativas del cliente.

Trazabilidad
- Cada requerimiento RF/RNF se mapea a al menos un test en cada categoría aplicable.
- Nomenclatura con ID facilita auditorías (ej: test_rq01_ingreso_sintoma).

Conclusión
La combinación de estos cuatro niveles asegura que el software es correcto internamente, funciona integrado, soporta escenarios reales y cumple formalmente lo pedido por el cliente.
